# Защита Minkert — что сделано и что делать вам

Этот документ описывает все уровни защиты проекта: что внедрено в коде, что настроить вручную, и какой эффект это даёт.

---

## Уровень 1 — API (сделано в коде)

### 1. Rate limiting (ограничение частоты запросов)

**Что:** `@nestjs/throttler` — лимиты по IP.

| Endpoint | Лимит | Зачем |
|----------|-------|-------|
| Весь API | 240 запросов/мин | Защита от DDoS и «залипших» скриптов |
| `POST /auth/login` | 10/мин | Защита от подбора пароля |
| `POST /auth/refresh` | 40/мин | Злоупотребление refresh-токенами |
| `POST /auth/bootstrap` | 3/час | Никто не создаст admin через интернет |
| Публичные дневники/задачи (запись) | 20–40/мин | Защита токен-ссылок от спама |

**Эффект:** брутфорс пароля и флуд API становятся практически бесполезными.

**Ваши действия:** ничего. Работает после деплоя.

---

### 2. JWT — проверка пользователя в базе

**Что:** при каждом запросе (с кэшем 60 сек) API проверяет, что пользователь ещё существует в БД.

**Зачем:** удалили аккаунт сотрудника — через минуту доступ пропадает, не через час.

**Эффект:** отозванные пользователи не ходят по API с старым токеном долго.

**Ваши действия:** ничего.

---

### 3. Bootstrap заблокирован в production

**Что:** `POST /auth/bootstrap` в production возвращает ошибку, даже если кто-то нашёл endpoint.

**Зачем:** нельзя создать второго «первого админа» через интернет.

**Эффект:** закрыта дыра первичной регистрации.

**Ваши действия:** ничего. `ALLOW_BOOTSTRAP=true` ставьте только при первом развёртывании на пустой базе, потом уберите.

---

### 4. Health check с проверкой базы

**Что:** `GET /api/health` делает `SELECT 1` в PostgreSQL.

- БД жива → `200 { ok: true, db: "up" }`
- БД мертва → `503 { ok: false, db: "down" }`

**Зачем:** Render видит реальные проблемы и может перезапустить сервис.

**Эффект:** меньше «зелёного» статуса при мёртвой базе.

**Ваши действия:** в Render убедитесь, что Health Check Path = `/api/health`.

---

### 5. Cron — напоминания и очистка токенов

**Что:** `@nestjs/schedule` внутри API:
- **каждую минуту** — напоминания о записях (не зависят от того, зашёл ли кто-то)
- **каждую ночь в 03:00** — удаление просроченных refresh-токенов

**Зачем:** раньше напоминания срабатывали только когда кто-то открывал уведомления.

**Эффект:** напоминания стабильны; БД не засоряется старыми сессиями.

**Ваши действия:** ничего (нужен always-on Render Starter).

---

### 6. Безопасные ошибки + Request ID

**Что:**
- В production клиент не видит stack trace — только «Внутренняя ошибка сервера»
- Каждый ответ содержит `requestId` (заголовок `X-Request-Id`)

**Зачем:** не светим внутренности; по ID можно найти ошибку в логах.

**Эффект:** меньше утечек информации атакующему.

**Ваши действия:** при обращении в поддержку сообщайте `requestId` из ответа API.

---

### 7. CORS — строгий режим в production

**Что:** без `CORS_ORIGIN` в production API не примет запросы с чужих сайтов.

**Зачем:** случайно не открыть API всему интернету.

**Эффект:** только ваш Vercel-домен может ходить в API.

**Ваши действия:** в Render **обязательно** задайте:
```
CORS_ORIGIN=https://ваш-сайт.vercel.app
```
(точный URL без слэша в конце; несколько доменов — через запятую)

---

### 8. Trust proxy + Helmet

**Что:** корректный IP за Render proxy; HSTS в production.

**Эффект:** rate limit считает реальный IP; браузер принудительно использует HTTPS.

**Ваши действия:** в Render уже добавлено `TRUST_PROXY=1` и `NODE_ENV=production` в `render.yaml`.

---

## Уровень 2 — CI (сделано в коде)

### GitHub Actions `.github/workflows/ci.yml`

**Что:** при каждом push в `main`:
- сборка backend + frontend
- unit-тесты auth
- e2e health check на **отдельной тестовой** PostgreSQL (не ваша Neon!)

**Зачем:** сломанный код не должен тихо попасть в прод.

**Эффект:** базовая сеть безопасности от регрессий.

**Ваши действия:**
1. Зайдите на GitHub → репозиторий → вкладка **Actions**
2. Убедитесь, что workflow **CI** зелёный после push
3. Если красный — не считайте деплой успешным, пока не почините

## Уровень 3 — что нужно сделать ВАМ (обязательно)

### A. Render — платный Starter (~510 ₽/мес)

| За что | Эффект |
|--------|--------|
| Always-on API | Нет 30–60 сек «просыпания» |
| Cron напоминаний | Работает 24/7 |
| Health check | Render перезапускает упавший инстанс |

**Действие:** Render Dashboard → `minkert-api` → Plan → **Starter**.

---

### B. Переменные окружения Render

| Переменная | Зачем | Пример |
|------------|-------|--------|
| `DATABASE_URL` | Подключение к Neon | pooled URL с `-pooler` |
| `JWT_ACCESS_SECRET` | Подпись токенов | `openssl rand -hex 32` |
| `CORS_ORIGIN` | Только ваш фронт | `https://minkert.vercel.app` |
| `NODE_ENV` | production-режим | `production` |
| `TRUST_PROXY` | IP за proxy | `1` |

**Действие:** Render → Environment → проверить все переменные.

---

### C. Neon — бэкапы (данные не трогаем)

| Способ | Как | Эффект |
|--------|-----|--------|
| Скрипт на Mac | `scripts/run-neon-backup.sh` | Локальная копия БД |
| Neon Console | Settings → Backups | Облачные бэкапы (платный план Neon) |

**Действие:** раз в неделю проверять, что бэкап создаётся. Раз в месяц — тест восстановления (`scripts/test-neon-restore.sh`).

**Важно:** индексы из `scripts/apply-safe-indexes.sql` — **безопасны**, данные не удаляют. Применяйте только после бэкапа, когда будете готовы.

---

### D. Секреты — ротация раз в год

| Секрет | Как сменить |
|--------|-------------|
| `JWT_ACCESS_SECRET` | Новый `openssl rand -hex 32` в Render → все пользователи перелогинятся |
| Пароли admin | Settings → учётные записи |

**Эффект:** даже при утечке старый секрет бесполезен.

---

### E. Проверка безопасности своего сайта

```bash
npm run security:check -- --url https://ваш-api.onrender.com --frontend https://ваш-сайт.vercel.app --yes
```

**Эффект:** автоматическая проверка HTTPS, заголовков, публичных endpoints.

**Действие:** запускайте раз в месяц или после крупных обновлений.

## Уровень 4 — рекомендуется (по желанию, сильный эффект)

### Sentry (мониторинг ошибок)

**За что:** письмо/push когда API падает с 500 в production.

**Действие:**
1. Зарегистрироваться на [sentry.io](https://sentry.io) (есть free tier)
2. Создать проект Node.js + React
3. Добавить `SENTRY_DSN` в Render (когда подключим в коде — пока вручную следите за логами Render)

**Эффект:** узнаёте о падении раньше сотрудников.

---

### Neon pooled connection

**За что:** стабильные соединения при нагрузке.

**Действие:** в Neon скопировать **Pooled** connection string → `DATABASE_URL` в Render, добавить `?sslmode=require`.

**Эффект:** меньше обрывов «too many connections».
